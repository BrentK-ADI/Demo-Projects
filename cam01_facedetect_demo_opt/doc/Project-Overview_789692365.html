1. Optimization by Example

Project Overview

The target project for this series is the Face Detection demo application running on the MAX78000 CAM01 demonstration hardware.  The demo continuously captures images from the on-board camera, passes the images through the loaded CNN, and displays an LED if a face was detected. All on a package the size of a thumbnail.

While the low power CNN is the highlight of this demo, the focus will be on the pre-processing and preparation of the image prior to loading it into the CNN accelerator.

The image pre-processing consists of two main algorithms, Auto White Balance/Color Correction, and Bayer Pattern Demosaicing.  In addition, the image loaded into the CNN accelerator needs to be rotated 90 degrees from the source. This is due to the CNN being training with portrait images, and our desire maximize the usable field-of-view of the onboard sensor.

Algorithm Flow

Preparing the Build

In the demo project, the existing color correct and debayering algorithms are located in the camera misc drivers package of the MSDK.  In order to provide flexibility in experimenting with the implementation, the debayering.c and debayering.h files were copied from the misc drivers into our project tree.  Additionally the project.mk file was modified to prevent including the MSDK's debayering.c in the build, and utilizing the local copy.

project.mk
### Remove the camera definition to override where debayering.c comes from
#CAMERA = HM0360_COLOR
#ifneq ($(BOARD),CAM01_RevA)
#$(error ERR_NOTSUPPORTED: This project is only supported on the MAX78000CAM01 board.  (see https://analog-devices-msdk.github.io/msdk/USERGUIDE/#board-support-packages))
#endif

#Manually add the camera driver, omitting the debayering.c from the misc drivers
SRCS += hm0360_color.c
PROJ_CFLAGS+=-DCAMERA_BAYER
PROJ_CFLAGS+=-DCAMERA_HM0360_COLOR

Configuring the Timer

In order to determine the impact of the algorithm changes, a timing mechansim must be used. For this project, an internal microsecond timer will be utilized to capture execution time for blocks of code. While using function calls to stop the timer and reading the register may introduce a few microseconds of additional time, versus toggling a GPIO and capturing the signal time, the algorithm measurement will be in the order of milliseconds, and this error will be negligible.

volatile uint32_t execTime;

MXC_TMR_SW_Start(MXC_TMR1);
/** Do work to be timed here **/
execTime = MXC_TMR_SW_Stop(MXC_TMR1);

Establishing a Baseline

Prior to modifying the algorithms to optimize for speed, a baseline of the standard implementation must be captured. Using the timer scheme above, the code execution time was measured prior to color correction (immediately after a full frame of camera data is loaded into RAM), up through loading all of the image data into the CNN accelerator (prior to running the CNN).

The total execution time as describe for a single frame is 98.849 ms.

Data Types Primer

There are several image data types used in this algorithm. Manipulation and conversion between these types will be covered as the algorithm is modified, however it is important to have a basic understanding of the formats and memory packing.

Bayer Pattern

The raw image data coming from the camera sensor is stored in 8-bit Bayer mosaic, or Bayer pattern format. For this format, each pixel of the image (in this examples case 320x240) is made up of a single 8-bit value.  The sensor itself is filtered, such that each pixel only represents either the Red, Green or Blue component, as shown below.  In order to establish the true life color of a given pixel, the values of the pixel and the surrounding pixels are interpolated to generate all three color components.  This interpolation process will be discussed more when evaluating the demosaicing algorithm implementation.  The Wikipedia Page on Bayer filtering provides a more detailed explanation.


RGB565

RGB565 is a color format which describes the color of a pixel within 16-bits of space. This allows for up to 65,536 possible colors.  The Red and Blue components utilize 5 bits, while the Green component utilized 6.  16-bit color is utilized by the color TFT displays that accompany the MAX32 series evaluation kits.

RGB888

RGB888, also known as 24-bit color, utilizes 3-bytes to describe a pixel, with 1 byte per Red, Green and Blue color channel. This is known as "True Color" and can represent 16M+ possible colors.  This color format is the most common among modern computer systems.
