Bayer: Iteration 7

This iteration of the Bayer algorithm will focus on reducing the number of memory accesses by utilizing a single memory read to obtain all necessary pixels for that given row.  If you recall, the AWB algorithm used a similar approach to conserve memory operations.  This was simple for the AWB, as the data within a given 32-bit was not dependent on any adjacent word.  The Bayer algorithm holds the challenge of required data sometimes being split across 2 32-bit words, dependent on the column being processed.  Take the figure below.  The bold black lines represent the 32-bit word boundaries in memory of our data.  From the figure, as processing works from column 0 through 3 (the magenta square), the memory requirements initially fall within a 32-bit boundary, as column 2 and 3 are processed, 2 32-bit words must be accessed to get the data.

Fixed Alignment Data

There is no escaping the need to access consecutive words for column 2 and 3. To meet the objective of minimizing memory accesses, there are several approaches which can be used in conjunction with the figure above, such as having separate conditional blocks depending on column number.  The approach this implementation will take is to generate word aligned data regardless of the column being accessed.  In this approach, data will be read from the source (either 1 or 2 words worth), and based on the column being processed, the left most pixel of the Bayer processing kernel will always be in the least significant byte of the created word. In addition to helping the memory access problem, by standardizing the location of data for the Bayer processing, when performing the averaging, all pixel access can occur the same way, regardless of column.  

Data Access

To support the new alignment, a function, FetchBayerValues is created.  This function takes in the pointer to the first word of data for the given row, as well as the column to be processed. Based on the column number, the data will be pulled from memory and aligned as described above. The value returned will utilize a new union to access individual components, leveraging  the pixel_data_t struct used in the AWB iterations.

static inline pixel_data_t FetchBayerValues(uint32_t* srcPtr, uint32_t column)
{
    pixel_data_t retVal;
    uint32_t row;
    uint32_t row2;
    row = *srcPtr;//Get the primary word
    switch(column & 0x3)
    {
        case 0:
            //Do Nothing
            break;
        case 1:
            row = row >> 8;
            break;
        case 2:
            row = row >> 16;
            row2 = *(srcPtr+1);
            row |= (row2 << 16);
            break;
        case 3:
            row = row >> 24;
            row2 = *(srcPtr+1);
            row |= (row2 << 8);
            break;        
            break;
    }
   
    retVal.word = row;
    return retVal;
}

Using The Data

The actual Bayer loop is modified to support the abilities provided by the new aligned data.  First, 3 variables (row0, row1, and row2) are defined to store the 3 aligned words representing each row of the Bayer kernel.  As the loops iterate and the kernel window moves upwards, rows0 and 1 and be reused, and stored in rows 1 and 2 respectively, at which point row0 is loaded from memory.  It is likely that rows0 and 1 will be pushed on the stack at some point, however the cost of popping the values off the stack should be significantly less than rereading possibly 2 memory values, checking the column and performing the shifts.

Secondly, since the FetchBayerValues function works from a uint32_t pointer, the address is maintained and updated in the loop, as opposed to using a helper function such as _i and accessing the source image as an array.  And finally, the actually interpolation is updated to use the fields from the pixel data_t type as needed.
